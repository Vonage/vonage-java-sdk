53    /**
54     * Instanciate a new message request.<br>
55     * This constructor exposes the full range of possible parameters and is not for general use
56     * Instead, it is accessed via super() in the constructors of various sub-classes that expose a relevant
57     * sub-set of the available parameters
58     *
59     * @param type the type of message will influence the makeup of the request we post to the Nexmo server, and also the action taken by the Nexmo server in response to this message
60     * @param from the 'from' address that will be seen on the handset when this message arrives,
61     *             typically either a valid short-code / long code that can be replied to, or a short text description of the application sending the message (Max 11 chars)
62     * @param to   the phone number of the handset that you wish to send the message to
63     * @param messageBody The text of the message to be sent to the handset
64     * @param binaryMessageBody The raw binary message data to be sent to a handset.
65     *                          This api, and the Nexmo messaging service will send this data 'as-is' (in conjunction with the binary UDH) and will not make any corrections.
66     *                          so you should ensure that it is a correctly constructed message
67     * @param binaryMessageUdh Most binary content will require a UserDataHeader portion of the message containing commands to enable the handset to interpret the binary data
68     *                         (for example, a binary ringtone, a wap-push, OverTheAir configuration, etc).
69     *                         Additionally, if you are sending a long text message as multiple concatenated messages and are performing this operation manually rather than
70     *                         using the automated long messaging handling in the Nexmo messaging service, then you will need to construct and include here an appropriate
71     *                         UserDataHeader field that describes the segmentation/re-assembly fields required to sucessfully concatenate multiple short messages.
72     * @param clientReference This is a user definable reference that will be stored in the Nexmo messaging records. It will be available in detailed reporting / analytics
73     *                        In order to help with reconcilliation of messages
74     * @param unicode This flag is set to true if the message needs to be submitted as a unicode message. This would be for scenario's where the message contains text that does not
75     *                fit within the Latin GSM alphabet. Examples would be messages to be sent in non-western scripts, such as Arabic, Kanji, Chinese, etc.
76     * @param statusReportRequired If set to true, then a delivery notification will be requested for this message delivery attempt.
77     *                             Upon receiving notification of delivery or failure from the network, the Nexmo platform will submit a notification to the url configured in your
78     *                             Nexmo REST account that represents the outcome of this message.
79     * @param wapPushUrl This is the url that will be submitted to the handset and will appear as a browsable message in the Inbox.
80     * @param wapPushTitle This is the title that will be associated with the url being submitted to the handset
81     * @param wapPushValidity This is the length of time (in seconds) that the message will be available for once delivered to the handset.
82     *                        Once this time has expired, the url will no longer be visible on the handset to be browsed (Subject to handset compatibility)
83     * @param messageClass (Optional) The message class that is to be applied to this message.
84     * @param protocolId The value of the GSM Protocol ID field to be submitted with this message. Ordinarily this should be left as the default value of 0
85     */
